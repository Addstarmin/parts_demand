CMD-X サプライチェーン需要予測・在庫最適化システム 発表用まとめ
======================================================================

1. システムの一言概要
--------------------

CMD-Xは、中部地区の中小部品メーカーを想定した、需要予測・BOM展開・在庫最適化・増減産内示シミュレーション・JIT出荷ピーク予測・実績データ連携をまとめて扱うサプライチェーン支援システムです。

単なるグラフ表示アプリではなく、

製品需要を予測する
-> BOMで構成部品需要へ展開する
-> 現在庫・安全在庫と比較する
-> 生産量・発注量・出荷量を推奨する
-> 完成車メーカーの増減産内示を反映する
-> JIT便のピークを予測する
-> 実績CSVを取り込み、予測精度と安全在庫を更新する

という、製造業の実務に近い一連の流れをデモできる構成になっています。


2. 解決したい課題
-----------------

製造業、特に部品メーカーでは、次のような課題が起きやすいです。

- 完成車メーカーからの内示変更により、需要が急に増減する
- 製品単位では需要を把握できても、部品単位の必要数まで即座に見えない
- 共通部品が複数製品で使われている場合、需要影響を見落としやすい
- 現在庫が多すぎるのか少なすぎるのか、属人的に判断されがち
- 安全在庫が固定値運用になり、予測誤差やリードタイム変動を反映できない
- JIT出荷の曜日・時間帯ピークが読めず、配送や現場負荷が集中する
- 実績データを更新しても、予測・評価・安全在庫へつながらない

CMD-Xはこれらを一つの画面とAPIでつなぎ、発表では「予測して終わり」ではなく「業務判断につながるAIシステム」として説明できます。


3. 主な利用者
-------------

想定ユーザーは、部品メーカーの以下の担当者です。

- 生産管理担当者
- 調達担当者
- 在庫管理担当者
- 営業・完成車メーカー対応担当者
- 工場管理者

画面上では、工場、製品、部品を選択しながら、需要予測、推奨発注、推奨生産、JIT出荷、内示影響、安全在庫を確認します。

8. 需要予測の仕組み
-------------------

このシステムの需要予測は、発表上は「Prophet 0.4 + XGBoost 0.6のアンサンブル思想」として説明できます。

ただし、デモ速度と環境安定性を優先するため、実装上は軽量な時系列ロジックで同じ考え方を再現しています。

Prophet相当の役割:
- 中長期トレンドを見る
- 年間季節性を見る
- 週番号ごとの傾向を見る
- 需要の緩やかな増減を捉える

XGBoost相当の役割:
- 直近需要のラグを使う
- 直近8週の移動平均を見る
- 短期変動を反映する
- 外部指標による補正を加える

アンサンブル:
- Prophet相当の予測を40%
- XGBoost相当の予測を60%
- これらを合成して最終予測を作る

式のイメージ:

final_forecast = prophet_like * 0.4 + xgb_like * 0.6

これにより、単純な平均やランダムな値ではなく、

- 過去の傾向
- 季節性
- 直近の変化
- 外部指標

を組み合わせた予測として説明できます。


9. 外部指標の扱い
-----------------

需要予測では、為替、PMI、気象のような外部指標も扱います。

主な外部指標:
- ドル円為替
- 製造業PMI
- 気温・天候

外部APIが使える場合は取得を試みますが、取得できない場合でもフォールバック値で動作します。

つまり、発表や授業環境でネットワークが不安定でも、システムが止まらない設計です。

外部指標の役割は需要を完全に決めるものではなく、あくまで弱い補正要因です。

例:
- PMIが高い -> 製造業全体が強く、需要を少し上げる
- 円安 -> 調達リスクや輸出関連影響として安全在庫や需要を少し補正
- 悪天候 -> 物流遅延リスクとして安全在庫を厚めに見る


10. BOM展開の仕組み
-------------------

BOMとは、製品を作るために必要な部品表です。

このシステムでは、製品需要を予測した後、BOMを使って部品需要へ変換します。

基本式:

部品必要数 = 製品予測数 × quantity_per_product

例:

PROD-A キャブレターASSYに、PT-1002 キャブレターボルトが2個必要な場合、

PROD-Aの来週予測が1,229個
PT-1002の使用数が2個

であれば、

PT-1002の必要数 = 1,229 × 2 = 2,458個

となります。

また、PT-1002やPT-5007のような共通部品は、複数製品からの需要が合算されます。

これにより、製品単位の増減産内示が、どの部品にどれだけ影響するかを説明できます。


11. 在庫・生産・発注・出荷推奨の仕組み
---------------------------------------

部品ごとの推奨量は、需要予測、現在庫、安全在庫から計算します。

基本式:

不足量 = 来週需要 + 安全在庫 - 現在庫

推奨生産量 = max(0, 不足量)
推奨発注量 = max(0, 不足量)
推奨出荷量 = 来週需要

意味:
- 来週需要を満たすだけでなく、安全在庫も維持したい
- 現在庫が十分なら追加生産・発注は0
- 現在庫が不足するなら、生産・発注を推奨
- 出荷推奨量は顧客需要と整合する

代表デモ対象:
- 工場: F-02 三河精密工場
- 製品: PROD-A キャブレターASSY
- 部品: PT-1002 キャブレターボルト

代表値:
- 来週部品需要: 2,458個
- 現在庫: 1,800個
- 安全在庫: 540個
- 生産推奨量: 1,198個
- 発注推奨量: 1,198個
- 出荷推奨量: 2,458個

この数値は、

2,458 + 540 - 1,800 = 1,198

という関係になっています。

つまり、画面上のKPIが独立したダミー値ではなく、業務式でつながっていることを説明できます。


12. JIT出荷ピーク予測の仕組み
-----------------------------

JITとは、必要なものを必要なタイミングで納入する考え方です。

このシステムでは、JIT出荷履歴から曜日・時間帯別の構成比を計算し、来週の出荷推奨量を各便に按分します。

対象便:
- 06:30
- 10:00
- 15:30
- 20:00

処理の流れ:

1. jit_shipment_history.csvを読む
2. 工場・部品ごとに絞り込む
3. 曜日と時間帯で過去出荷量を集計
4. 各スロットの構成比を計算
5. 来週出荷量を構成比で按分
6. 丸め誤差は最大構成比の便で補正

代表デモでは、F-02 / PT-1002で火曜10:00が最大ピークになります。

代表値:
- 火曜10:00: 233個
- 週間JIT合計: 2,458個

JITのポイント:
- 各便の合計は週次出荷量と一致
- ピーク便が明確に見える
- 配送便の事前予約や現場人員配置の説明につながる


13. 完成車メーカー内示シミュレーション
-------------------------------------

完成車メーカーから「来月は20%減産」「来週は15%増産」のような内示変更が来ることがあります。

このシステムでは、メーカー、製品、調整率を指定すると、通常予測と調整後予測を比較できます。

計算式:

100%紐づきの場合:
V_adjusted = round(V_normal × (1 + r / 100))

一部配分の場合:
V_adjusted = round(V_normal × (1 - allocation) + V_normal × allocation × (1 + r / 100))

例:

完成車A社、PROD-A、-20%の場合、

通常予測: 1,229個
調整後予測: 983個

となります。

さらに、製品需要の変化はBOMを通じて部品需要へ展開されます。

つまり、

完成車メーカーの内示変更
-> 製品需要の変化
-> BOMによる部品需要の変化
-> JIT出荷量の変化
-> 生産・発注判断の変化

までを一連の流れとして見せられます。


14. 動的安全在庫の仕組み
-----------------------

安全在庫とは、需要のブレや納期遅れに備えて持っておく在庫です。

従来の固定安全在庫では、需要予測が外れやすい部品も、安定している部品も同じように扱ってしまいます。

CMD-Xでは、予測誤差RMSEと調達リードタイムから安全在庫を計算します。

基本式:

RMSE = sqrt(mean((actual_demand - predicted_demand)^2))

dynamic_safety_stock = ceil(safety_factor × RMSE × sqrt(lead_time_days / 7))

意味:
- 予測誤差が大きいほど安全在庫を増やす
- リードタイムが長いほど安全在庫を増やす
- 予測精度が改善した部品は安全在庫を減らせる
- リードタイムが短くなった部品も安全在庫を減らせる

安全係数:
- デフォルトは1.65
- サービスレベル95%相当の考え方

ガードレール:
- 最小安全在庫
- 最大安全在庫
- 1回の変更幅を最大50%に制限
- 20%以上の変化はレビュー対象

代表デモ:
- PT-1002は予測精度改善またはLT短縮により、安全在庫が540個から270個へ減少するストーリー
- 別部品ではRMSE増加やLT長期化により、安全在庫が増加する例も用意


15. データ管理・実績連携の仕組み
-------------------------------

このシステムは、デモデータを見るだけでなく、実績CSVを取り込んで予測へ反映できます。

流れ:

1. 現在の実績CSVをダウンロード
2. Excelなどで実績を編集
3. CSVをアップロード
4. サーバー側で検証
5. エラーがなければ確定反映
6. 反映前にバックアップ作成
7. CSVを原子的に更新
8. 必要に応じて予測再計算
9. 必要に応じて安全在庫再計算
10. 更新履歴をSQLiteへ保存

更新方式:
- append: 追記
- upsert: 追記・上書き
- replace: 全件置換

replaceは危険なため、画面上で確認チェックが必要です。

CSV検証:
- 必須カラム
- 日付形式
- 数値型
- 負数
- 0禁止列
- 主キー重複
- ID参照整合性
- CSV数式インジェクション警告
- 極端な外れ値警告

バックアップ:
- backend/data/backups/ に保存
- manifest.jsonつき
- SQLite DBも必要に応じてバックアップ
- 復元前にも現在状態をバックアップ

これにより、発表では「企業が持っている実績CSVを取り込んでAI予測へ反映できる」と説明できます。


16. 週次自動更新の仕組み
-----------------------

実運用では、基幹システムや生産管理システムから毎週実績データを取得する想定です。

今回のデモでは、2種類の更新ソースを実装しています。

1. demo source
   - 現在CSVの最終実績週を確認
   - 次週を算出
   - 製品需要、部品実績、JIT出荷、予測精度履歴を疑似生成
   - BOMと在庫式の整合性を保つ
   - 同じ週を重複追加しない

2. directory source
   - 指定ディレクトリに置かれたCSVを読み込む
   - 同じCSV検証を通す
   - 成功時はprocessedへ移動
   - 失敗時はfailedへ移動し、エラーログを残す

APSchedulerを使うことで、毎週月曜6時のような定期更新に拡張できます。

設定は.envをブラウザから書き換えるのではなく、SQLiteに保存します。


17. SQLiteの役割
----------------

CSVはマスタ・時系列実績データを保持します。

SQLiteは、操作履歴や設定の保存に使います。

保存する主な内容:
- 増減産内示シミュレーション履歴
- 安全在庫変更履歴
- 安全在庫設定
- データ更新履歴
- 週次自動更新設定

つまり、CSVは業務データ、SQLiteは操作履歴・設定データという役割分担です。


18. デモデータの特徴
-------------------

デモデータは完全ランダムではなく、製造業の業務関係を反映した半構造化データです。

需要データ:
- 長期トレンド
- 年間季節性
- 四半期末需要
- 年末年始、ゴールデンウィーク、お盆の低下
- 連休前の先行生産・先行出荷
- 製品ごとの差別化
- 小さなノイズ

在庫データ:
- 期首在庫 + 生産 - 出荷 + 棚卸差異
- 需要と無関係な乱数ではない
- 適正在庫、やや不足、やや過剰など複数状態を含む

JITデータ:
- 火曜10:00が代表ピーク
- 木曜15:30が第2ピーク
- 月曜は多め、金曜は少なめ、土日は少量

安全在庫:
- 減少する部品
- 増加する部品
- ほぼ変わらない部品

これにより、グラフやKPIを説明しやすくしています。

20. 発表での説明例
-----------------

このシステムは、製造業の部品メーカーが、需要予測から在庫・発注・出荷判断までを一気通貫で確認するためのプロトタイプです。

従来は、製品需要、部品表、現在庫、安全在庫、JIT出荷予定、メーカー内示が別々に管理されているため、需要が変わったときにどの部品がどれだけ不足するかを判断するのに時間がかかります。

CMD-Xでは、まず製品需要をAIで4週間予測します。次にBOMを使って部品ごとの必要数に展開します。そのうえで、現在庫と安全在庫を比較し、生産・発注・出荷の推奨量を表示します。

さらに、完成車メーカーからの増減産内示を入力すると、通常予測と調整後予測の差が製品・部品・JIT出荷に反映されます。

安全在庫については、固定値ではなく、予測誤差RMSEとリードタイムから動的に再計算します。予測精度が改善した部品は安全在庫を減らし、誤差やリードタイムが増えた部品は安全在庫を増やすことができます。

また、データ管理画面から企業が保有する実績CSVを取り込み、予測や安全在庫へ反映できます。実運用では、基幹システムや生産管理システムから毎週実績を取得する運用へ拡張できます。


21. 機械学習としてのポイント
---------------------------

このシステムの機械学習・AI要素は、単に未来の数字を出すことではありません。

重要なのは、予測値を業務判断へつなげている点です。

AI要素:
- 時系列予測
- トレンド・季節性の把握
- 直近ラグ特徴の反映
- 外部指標による補正
- 予測精度評価
- RMSEに基づく安全在庫最適化

業務接続:
- 予測値をBOM展開
- 在庫と比較
- 生産・発注・出荷推奨へ変換
- JIT便へ配分
- 内示シミュレーションに反映
- 実績取り込みで評価・改善

つまり、「予測AI」ではなく「予測を使って意思決定するシステム」として説明できます。


22. このシステムの強み
---------------------

1. 需要予測から部品需要までつながっている
2. BOMにより共通部品の影響も見える
3. 現在庫・安全在庫と比較して推奨量を出せる
4. 完成車メーカーの内示変更をすぐに反映できる
5. JIT出荷ピークが見える
6. RMSEとリードタイムで安全在庫を動的に更新できる
7. CSV取り込みにより、実績データ更新の流れをデモできる
8. バックアップ・復元があり、データ更新に安全性がある
9. FastAPIとReactで実際に動くプロトタイプになっている
10. 発表で数値の関係を説明しやすい


23. 実行方法
------------

デモデータ再生成:

python3 -m scripts.generate_demo_data

デモデータ検証:

python3 -m scripts.validate_demo_data

バックエンド起動:

cd backend
python3 -m uvicorn main:app --reload

フロントエンド起動:

cd cmdx-frontend
npm run dev

バックエンドテスト:

cd backend
pytest

フロントエンド確認:

cd cmdx-frontend
npm run lint
npm run build


24. 最後に伝えるとよいまとめ
---------------------------

CMD-Xは、製造業の需要予測を「予測グラフ」だけで終わらせず、BOM、在庫、生産、発注、出荷、JIT、内示、安全在庫、実績連携までつなげたシステムです。

発表では、次の一文で締めると伝わりやすいです。

「このシステムは、AIで需要を予測するだけでなく、その予測を使って、どの部品を、いつ、どれだけ作り、発注し、出荷すべきかまで判断できる、製造業向けの需要予測・在庫最適化システムです。」
